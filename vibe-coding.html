The 3 Phases to Vibe Coding

Before Building

• Interrogate your idea

• Write 6 canonical docs

• Write CLAUDE.md with rules

• Create progress.txt

• Gather UI screenshots

• Init git, push to GitHub

While Building

• AI reads CLAUDE.md + progress.txt first

• Plan mode before coding

• One feature at a time

• Reference docs in every prompt

• Commit after each working feature

• Update progress.txt + lessons.md

Before Shipping

• Check mobile

• Check error & empty states

• Verify secrets are hidden

• Test main user flow end-to-end

• Check performance

• Ship, then iterate on feedback

Before Building

Set up your system before writing a single line of code

Created by Nicole Garcia Fischer

Documentation-First, Code Second

Write these 6 canonical docs before touching any code. They cross-reference each other and give AI everything it needs.

PRD.md

Full spec: what you’re building, who it’s for, features, scope, success criteria. Your contract with AI.

APP_FLOW.md

Every page, every user path. Step-by-step sequences with decision points, success/error states.

TECH_STACK.md

Every package locked to exact versions. Eliminates hallucinated dependencies and random tech choices.

FRONTEND_GUIDELINES.md

Complete design system: fonts, colors (hex), spacing scale, component styles, responsive breakpoints.

BACKEND_STRUCTURE.md

Database schema, auth logic, API endpoint contracts, storage rules, edge cases.

IMPLEMENTATION_PLAN.md

Step-by-step build sequence. The more steps, the less AI guesses. The less it guesses, the fewer hallucinations.

You’re not writing docs for humans. You’re writing constraints for AI. Markdown is the language AI thinks in.

Created by Nicole Garcia Fischer

The LLM Is Your Coworker, Not Your Tool

When you frame AI as a team member, everything changes. — Randall Bennett

Tool Mindset

− Press button, get output

− Blame it when output is bad

− No context, no feedback

− Stays at P50 (median)

Coworker Mindset

+ Explain goals like you would to a colleague

+ Give feedback like a junior developer

+ Build team culture through docs & prompts

+ Walk it up to P90 for your domain

The P50 → P90 shift: By default, an LLM operates at the 50th percentile of humanity. Your job is to walk it up to the 90th percentile for your specific domain through context, culture, and intent.

The Intent.md Framework

Randall’s 7-section file for every project. Create at the company, project, or feature level. Even a rough one beats having none.

1

Purpose

Why are we building this? What’s the reason anything is happening?

2

End State

What does success look like? How will this feel when it’s done?

3

Constraints

What’s already decided? Non-negotiable boundaries AI must respect.

4

Trade-offs

Ranked priorities. “Feature parity over abstraction purity.” What to sacrifice for what.

5

Risk Tolerance

Where can AI be bold? Where must it be cautious? Where is breakage acceptable?

6

Escalation Conditions

When should AI stop and ask you? Define the hard stops.

7

Verification Steps

How do we know it worked? Specific tests, checks, or manual confirmations.

The Interrogation System

Before you even write documentation, make AI tear your idea apart. AI hallucinates when your clarity ends — so extend your clarity.

"Before writing any code, endlessly interrogate my idea in Planning mode only. Assume nothing. Ask questions until there are no assumptions left."

What AI should ask you:

• Who is this for?

• What’s the core action?

• What happens on success?

• What data gets saved?

• What happens on error?

• Does this need auth?

• Does this need a database?

• Does it work on mobile?

• What data gets displayed?

Then generate your docs from the answers:

"Based on our interrogation, generate my canonical documentation files: PRD.md, APP_FLOW.md, TECH_STACK.md, FRONTEND_GUIDELINES.md, BACKEND_STRUCTURE.md, IMPLEMENTATION_PLAN.md. Be specific and exhaustive."
Interrogation → Documentation → Code

While Building

Execute with discipline and iterate

Created by Nicole Garcia Fischer

Self-Improving CLAUDE.md

CLAUDE.md is a living document, not a static file. It gets better every session.

After every correction, end with: “Edit CLAUDE.md so you don’t make that mistake again.”

Create a lessons.md file. After every debugging session, Claude updates it with the pattern that caused the issue + the rule that prevents it.

Point CLAUDE.md at it: “Review lessons.md at session start.”

# CLAUDE.md (self-improving) ## Rules - Use bun, not npm - Never use inline styles - Mobile-first responsive ## References Read @docs/prd.md for requirements Read @docs/lessons.md at session start Read @progress.txt for current state ## Learned corrections - Always check edge runtime for /api - Use tenant ctx for billing lib - Run typecheck before committing

Claude is eerily good at writing rules for itself. Over time, your mistake rate drops because AI is encoding its own corrections.

Created by Nicole Garcia Fischer

progress.txt: Your Session Bridge

AI has no memory between sessions. When you close a terminal, everything is gone. progress.txt is your external memory.

# progress.txt COMPLETED: - User auth via Clerk (login, signup, Google OAuth) - Dashboard layout with sidebar navigation - Products API endpoint (GET /api/products) IN PROGRESS: - Product detail page (/products/[id]) - Need to connect frontend to API NEXT: - Shopping cart functionality - Checkout with Stripe KNOWN BUGS: - Mobile nav doesn't close after clicking a link

• Update after every completed feature

• AI reads this at the start of every session

• Commit to git alongside your code — both GitHub and your context stay in sync

Created by Nicole Garcia Fischer

Specificity Compounds

The more you define upfront, the less you debug later. Same idea, completely different output quality.

Vague prompt

"Build me an app where users can post things"

Specific prompt backed by docs

"Read CLAUDE.md and progress.txt first. Then build step 4.2 from IMPLEMENTATION_PLAN.md. The login flow is in APP_FLOW.md section 3. Style per FRONTEND_GUIDELINES.md. Match the UI to the screenshot."

The specificity isn’t extra work — it is the work. Precision compounds. Vagueness compounds too, just in the wrong direction.

Investigation Mode vs Explicit Mode

Investigation

(YOLO mode)

• Back and forth with AI, exploring

• You’ll throw most of this away

• Figuring out what to build

• Learning, brainstorming, prototyping

Explicit

(Intentful mode)

• Set AI loose, it builds exactly what you expect

• One-shot or three-shot execution

• You know what you want

• Your process is solid and repeatable

Start here: Investigation → Graduate to: Explicit
Created by Nicole Garcia Fischer

The Debugging Loop

Errors are instructions, not insults. Vibe coding is iteration — the skill is iterating quickly.

1

Read the error

Actually read it. Don’t panic.

2

Find the location

What file, what line number?

3

Understand the claim

What does the error say is wrong?

4

Check the obvious

Typos, missing imports, wrong variable names

5

Give AI full context

Error message + code at that line + what you expected to happen

The loop: AI gives you code → you try it → it breaks → you paste the error → AI fixes → repeat. For stubborn bugs, switch to Debug mode or use subagents.

The After Action Report (AAR)

After every task, have AI write a report. Review AARs instead of reading every line of code. — Randall Bennett

# AAR: Add Stripe checkout ## What happened Implemented checkout flow with Stripe Elements. Added /api/checkout endpoint and webhook handler. ## Changes made - src/app/checkout/page.tsx (new) - src/app/api/checkout/route.ts (new) - src/app/api/webhooks/stripe/route.ts (new) - src/lib/stripe.ts (new) ## Decisions Used Stripe Elements instead of Checkout Sessions for more UI control per FRONTEND_GUIDELINES.md. ## Why Checkout Sessions would redirect users off-site, conflicting with the single-page flow in APP_FLOW.md.

“The AAR gives me everything I need. It tells me what files changed, why decisions were made, so I can feel comfortable this engineer is doing things correctly.”

— Randall Bennett

Pre-Ship Checklist

Don’t ship until you’ve answered every one of these. Your users will find every bug you missed.

☐

Mobile

Actually open it on your phone. Not just browser devtools.

☐

Empty states

What happens with no data? Is there a helpful message?

☐

Error states

What happens with wrong data? Does it fail gracefully?

☐

Loading states

What happens with slow internet? Is there feedback?

☐

Secrets hidden

API keys not in frontend code or browser devtools?

☐

Main flow works

Can a user complete the core action end-to-end?

☐

Stress test

Can you break it by clicking fast or submitting twice?

☐

Cross-browser

Does it work in Chrome, Safari, and Firefox?

Product Command

From military “Mission Command” — top-down on intent, bottoms-up on execution. — Randall Bennett

Shared Picture

The LLM understands what you’re building and why. Your docs, intent, and CLAUDE.md paint the full picture.

Decentralized Execution

AI can make reasonable decisions without you babysitting every step. It has enough context to act independently.

Disciplined Initiative

If AI makes a choice you didn’t expect, your intent wasn’t clear enough. Fix the docs, not the code.

Peacetime

Building systems, processes, docs, infrastructure. This is your prep work.

Wartime

Executing and shipping. You can only do this well if your peacetime prep is solid.

The Agent as Your Only Interface

Everything is code. Everything is an API. The agent should be your single interface to all tools. — Sid Bharath

Gmail Calendar Slack Database GitHub Analytics

Daily example: /plan-day

Checks calendar, reads emails, reviews yesterday’s incomplete tasks, proposes today’s priorities — all in one command.

Work example: “Prep my meeting”

Searches emails, finds deal status, identifies sticking points, suggests talking points — no app switching.

Turn Claude Code into Claude Teacher

Every project is a lesson to become more technical. Add this to your CLAUDE.md and Claude will write you a personalized learning doc after every project.

# Add this to your CLAUDE.md For every project, write a detailed FOR_[yourname].md file that explains the whole project in plain language. Explain: - The technical architecture - The structure of the codebase and how the various parts are connected - The technologies used and why we chose them - Lessons I can learn from it: bugs we hit and how we fixed them, pitfalls and how to avoid them, new technologies used, how good engineers think and work, best practices Make it engaging to read. Use analogies and anecdotes. Don’t make it sound like boring technical documentation.

The people who learn while they build are the ones who eventually stop needing guides. This turns every project into a curriculum — personalized to exactly what you just built.

— Zara Zhang (@zarazhangrui)

Vocabulary & Concepts

The words and ideas that make your prompts precise

Created by Nicole Garcia Fischer

Speak the Language — Code

Components = LEGOs

Reusable pieces of interface you snap together.

A button is a component. A navbar is a component. A card is a component. A form is components inside components.

“Build a landing page with: navbar, hero, features grid (3 cards), CTA, footer.”

State = Data That Changes

Why your app feels alive — or doesn’t.

Is the menu open or closed? Is the user logged in? What’s in the cart? When your button doesn’t work, it’s usually a state problem.

“When user clicks this button, set modal state to open.”

Layout = Boxes Inside Boxes

Master this and you understand 90% of web design.

Header at top, main content in middle, optional sidebar, footer at bottom. Sections, containers, grids, cards.

“Two-column layout. Sidebar 250px, fixed. Main content takes the rest.”

Pages & Routes

Page = what you see. Route = the URL that shows it.

Static: /about always shows the same page. Dynamic: /products/[id] shows different products.

“4 pages: home (/), about, products, single product (/products/[id])”

Created by Nicole Garcia Fischer

Speak the Language — Architecture

Frontend

What users see
(runs in browser)

↔

API

The waiter
GET / POST / PUT / DELETE

↔

Backend

Behind the scenes
(runs on server)

Database

Where you save stuff permanently. Without one, everything resets on refresh.

Need one? Users create accounts, save content, or data grows over time. Don’t? Static site, portfolio, no user interaction.

Authentication

Login/logout — proving who someone is. Harder than it looks.

Don’t build from scratch. Use Clerk (simplest) or Supabase Auth (if already using Supabase for DB).

Tell AI which you need upfront. “Frontend only, just a landing page” vs “Needs a backend with user accounts and a database” produce completely different outputs.

Created by Nicole Garcia Fischer

Design Vocabulary

UI = what it looks like (colors, fonts, spacing). UX = how it feels to use (intuitive? frustrating?). Be specific with AI about which you mean.

Design styles to reference in prompts

Glassmorphism

Frosted glass, blur, translucent. Apple/Spotify vibes.

Neobrutalism

Bold, thick borders, clashing colors. Gumroad energy.

Bento Grid

Modular blocks, varying sizes. Apple popularized this.

Dark Mode

Not a toggle — a design system. Plan both palettes from the start.

Micro-interactions

Button scales on hover, checkbox bounces. Separates polished from amateur.

Neumorphism

Soft, extruded 3D feel. Elegant but tricky for accessibility.

Don’t say “make it look modern.” Say “glassmorphism cards with bento grid layout, dark mode, and micro-interactions on hover states.” Lock design decisions in FRONTEND_GUIDELINES.md.

Created by Nicole Garcia Fischer

Screenshot-Driven Development

Visual references eliminate ambiguity faster than any written description. AI can analyze images and extract design patterns, spacing, colors, and component structures.

Reference a Design

Find an app you love on Dribbble or Framer, screenshot it, paste it into Claude Code.

“Match this layout and use this UI as inspiration.”

Fix Your WIP

Screenshot your own work in progress and show what’s wrong.

“Here’s what it looks like now. Here’s what’s wrong. Fix it.”

Compare Output

Paste the design and your implementation side by side.

“Compare my implementation to this screenshot. What’s different?”

On Mac: Ctrl+V to paste screenshots directly into Claude Code. Infinitely better than describing a design with words.

Created by Nicole Garcia Fischer

How to Iterate

Bad iteration

"It doesn't look right, fix it." "It's broken." "Make it better."

Good iteration

"The product grid shows 4 columns on desktop but I need 3. Card images are stretched — use object-cover. No loading state when data is fetching."

Break big ideas into small pieces. Think LEGOs. Each piece is one task, tested independently. “Build step 5 from IMPLEMENTATION_PLAN.md” not “build the next thing.”

The loop: AI builds v1 → you test → describe what’s wrong specifically → AI fixes → test again → repeat. First output is rarely right. That’s fine.

Created by Nicole Garcia Fischer

When AI Is the Wrong Tool

Use AI for

+ Generating boilerplate code

+ Writing repetitive logic

+ Exploring approaches quickly

+ Debugging with context

+ Translating your intent into code

Learn yourself

• Core concepts (everything in this deck)

• How to read code AI generates

• How to spot when AI is wrong

• How to debug when AI can’t help

• How your chosen stack works

If you rely on AI for everything, you’re building on quicksand. One weird bug and you’re stuck. Official docs are often better than AI — knowing how to search them is the fallback skill nobody talks about.
