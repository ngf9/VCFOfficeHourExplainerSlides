What is Claude Code?

An agentic coding tool that reads your codebase, edits files, runs commands, and integrates with your development tools — in your terminal, IDE, desktop app, and browser.

Capability Description
Build features & fix bugsDescribe in plain language, Claude implements across files
Automate tedious workTests, lint fixes, merge conflicts, dependency updates
Commits & PRsStages changes, writes messages, opens pull requests
Connect tools (MCP)Slack, Jira, databases, Google Drive
CustomizeCLAUDE.md, skills, hooks — persistent instructions & reusable workflows
Agent teamsMultiple agents working on subtasks in parallel

How Claude Code Works

Claude Code is an agentic assistant. It doesn’t just respond — it gathers context, takes action, and verifies results in a loop until your task is done.

Source: code.claude.com/docs

Built-in Tools

File opsRead, edit, create, rename files
SearchFind files, regex search, explore code
ExecutionShell commands, tests, git, builds
WebSearch, fetch docs, look up errors

What Claude Can Access

• Your project — all files in your directory

• Your terminal — any command you can run

• Git state — branch, uncommitted changes, history

• CLAUDE.md — your persistent instructions

• Auto memory — learnings Claude saves itself

• Extensions — MCP, skills, subagents, plugins

Best Way to Work with Claude Code: Go Slow to Go Fast

Invest in exploring and planning thoroughly before writing a single line of code.

1

Explore

Hit Shift+Tab for Plan Mode, read files, understand the codebase

2

Plan

Ask Claude for a detailed implementation plan

3

Implement

Switch to Normal Mode, let Claude code against the plan

4

Commit

Descriptive message, open a PR

Go back and forth with Claude until you like the plan, then switch to auto-accept.

Skip planning for trivial tasks — typos, single-line changes, simple renames.

01

The CLAUDE.md File

Persistent context for every session

Created by Nicole Garcia Fischer

Your AI’s Instruction Manual

What it is: a markdown file Claude reads at the start of every session — persistent context across conversations

What goes in it: coding standards, project architecture, workflow rules, gotchas — things Claude can’t infer from code

What makes a great one: specific > vague. Keep under ~200 lines. Every line earns its place.

Scoping: project-level (shared via git), user-level (personal prefs), or org-wide

# CLAUDE.md ## Code style - Use bun, not npm - Prefer type over interface - ES modules only ## Workflow - Typecheck after changes - Run single tests, not full suite ## Gotchas - /api uses edge runtime - Billing lib needs tenant ctx

Run /init to generate a starter CLAUDE.md, then refine over time.

Created by Nicole Garcia Fischer

What Goes in CLAUDE.md

Include

  • + Verification steps — how Claude checks its own work
  • + Bash commands Claude can’t guess
  • + Code style rules that differ from defaults
  • + Testing instructions & preferred runners
  • + Repo etiquette (branch naming, PR conventions)
  • + Architectural decisions & common gotchas

Skip

  • − Anything Claude figures out from code
  • − Standard language conventions
  • − Detailed API docs (link instead)
  • − Info that changes frequently
  • − File-by-file codebase descriptions
  • − Self-evident practices (“write clean code”)

You can place CLAUDE.md files in several locations:

• Home folder (~/.claude/CLAUDE.md): applies to all Claude sessions

• Project root (./CLAUDE.md): check into git to share with your team

• Parent directories: useful for monorepos where both root/CLAUDE.md and root/foo/CLAUDE.md are pulled in automatically

• Child directories: Claude pulls in child CLAUDE.md files on demand when working in those directories

Verification is the single highest-leverage thing you can include.

Tests, screenshots, linters, bash commands — give Claude a way to check its own work and quality goes up 2–3x.

Managing Context Beyond CLAUDE.md

Keep CLAUDE.md lean (~200 lines) — it’s an index, not a monolith

Offload detailed context into separate files: docs/prd.md, docs/design-system.md, docs/api-spec.md

Use @path imports in CLAUDE.md so Claude reads them when relevant

CLAUDE.md tells Claude when to reference each doc — the docs contain the what

# CLAUDE.md ## Project context Read the PRD for feature requirements: @docs/prd.md ## Design Follow the design system for all UI work: @docs/design-system.md ## API Reference the API spec before adding endpoints: @docs/api-spec.md ## Code style - Use bun, not npm - Prefer type over interface

This pattern keeps CLAUDE.md concise while giving Claude deep context on demand. The referenced files load only when Claude needs them.

Skills: Reusable Playbooks for Claude

Skills extend Claude’s knowledge with project-specific workflows. Here’s a real one from the Anthropic docs:

# .claude/skills/fix-issue/SKILL.md --- name: fix-issue description: Fix a GitHub issue disable-model-invocation: true --- Analyze and fix the GitHub issue: $ARGUMENTS 1. Use `gh issue view` to get details 2. Search the codebase for relevant files 3. Implement the fix 4. Write and run tests to verify 5. Create a descriptive commit message 6. Push and create a PR

Type /fix-issue 1234 and Claude handles the rest.

Skills can also inject live data with !`gh pr diff` — shell output gets pulled in before Claude sees the prompt.

Building a Great Skill

Don’t State the Obvious

Claude already knows a lot about coding. Focus on information that pushes it out of its defaults — your org’s unique patterns, not standard conventions.

Build a Gotchas Section

The highest-signal content in any skill. Add common failure points as Claude hits them — this is what makes a skill get better over time.

Use the File System

Skills are folders, not just markdown. Include scripts, references, examples, and templates. Claude discovers and uses them at the right time.

Description = Trigger

The description field is for the model, not humans. It’s what Claude scans to decide “is there a skill for this?” Write it as a trigger condition.

You don’t need to create all skills from scratch.

MCP: Connect Claude to External Tools

Claude Code’s built-in tools handle files, search, and execution — but can’t reach external services. MCP is the open standard that bridges that gap.

Configure via .mcp.json in your project root. Shared with the team via git.

Databases

Query & update

Slack

Search & post

Jira

Tickets & boards

Google Drive

Docs & sheets

Figma

Design files

Browser

Control & test

MCP vs Skills — when to use which?

MCP = Claude needs to interact with an external system (query, post, control). Without MCP, Claude simply can’t reach it.

Skills = Claude needs knowledge about how to use those systems (schemas, patterns, workflows). Zero token cost until invoked.

MCP tool definitions load at session start and consume context every request. Disconnect servers you’re not actively using (/mcp to check costs).

Run claude mcp add to connect external tools like Notion, Figma, or your database.

Additional Tips

Permissions

Pre-allow safe commands via /permissions so Claude doesn’t ask every time:

Bash(bun run build:*) Bash(bun run test:*) Bash(gh pr diff:*)

Parallel Sessions

Run multiple Claude sessions simultaneously, each in its own git worktree. Work on several features at once without conflicts.

Hooks

Deterministic scripts at lifecycle events — auto-format after edits, lint before commits, log on session end.

{ "matcher": "Write|Edit", "hooks": [{ "command": "bun run format" }] }

REF: Tips & Tricks

Models & Think Modes

Switch models with /model. Trigger deeper reasoning by adding keywords to your prompt:

think → extended thinking • think hard → deeper analysis • ultrathink → maximum reasoning

Opus for complex architecture • Sonnet for fast iteration • Haiku for quick lookups

Paste Images & Screenshots

Claude can read screenshots, mockups, error messages, and diagrams — great for debugging UI or implementing designs.

Mac: Ctrl+V to paste from clipboard • Or drag & drop image files directly into the prompt

Custom Status Line

See model, branch, and context window usage at a glance:

/status-line show model, branch, context window usage

Example output:

opus | MyProject git:(main) x | ████ 22%

More Tricks

• @file to reference files in your prompt

• Esc to interrupt Claude mid-response
